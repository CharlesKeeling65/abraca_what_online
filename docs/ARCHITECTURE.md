# 架构设计（跨端 + 远程联机 + 自建服务器）

## 1. 总体架构

采用 **Authoritative Server（服务端权威）**：

- 客户端只发送“意图”（例如施放咒语、结束回合）。
- 服务端负责规则校验、随机数（骰子）、状态推进、结算广播。
- 客户端只渲染服务器下发的状态快照与事件流。

这样可避免作弊，并保证移动端/PC 端一致性。

## 2. 工程分层

1. **共享层（`packages/shared`）**
   - 游戏常量、数据结构、协议、状态机与规则引擎。
2. **服务层（`apps/server`）**
   - 房间生命周期、玩家连接管理、指令转发、状态广播。
3. **客户端层（`apps/web` / `apps/mobile` / `apps/desktop`）**
   - UI、交互输入、动画、断线重连。

## 3. 联机模型

- 支持房间：创建房间 / 加入房间 / 观战（后续可扩展）。
- 房间状态：`waiting -> in_round -> scoring -> finished`。
- 断线恢复：客户端重连后请求 `RESYNC`，服务器返回最新快照。

## 4. 自建服务器部署

- 提供 Dockerfile + docker-compose 模板。
- 服务器可在公网或内网部署；客户端通过 `WS_URL` 连接。

## 5. 跨端建议

- Web：React/Vite。
- Mobile：React Native（Expo）复用协议与状态管理。
- Desktop：Tauri 包装 Web 客户端，复用同一前端代码。

## 6. 开发里程碑

1. M1：共享模型 + 服务端房间与基础协议。
2. M2：完整回合规则判定与单元测试。
3. M3：Web MVP（创建/加入房间 + 对局）。
4. M4：移动端适配 + 桌面打包。
5. M5：匹配、观战、回放、排行榜。
